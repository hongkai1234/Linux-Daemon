Linux 核心守護神 Daemon

深入了解 Ubuntu 作業系統核心，探索那些默默支撐系統運行的後台力量。

1. 什麼是 Daemon？

Daemon（發音同 "demon"）是指在後台運行、不與用戶直接進行交互的計算機程序。

在 Linux 檔案系統中，你常會看到以 "d" 結尾的程序名稱，這通常就代表它是一個 Daemon。例如：

  • sshd：負責遠端連線的 Secure Shell Daemon。
  • httpd：負責網頁服務的 HTTP Daemon。
  • systemd：系統啟動後的第 1 個進程，負責管理所有其他 Daemon。
"想像它是 Ubuntu 系統的自動化管家，你不需要叫它，它也會主動幫你處理雜事。"

2. Daemon 的核心特點

後台運行

它沒有圖形界面，也不會佔用你的終端機輸入窗口。你看不見它，但它確實在那工作。

生命週期長

通常在系統啟動（Boot）時自動開啟，直到系統關閉（Shutdown）時才停止，始終守護系統。

非交互式

不需要手動輸入指令來維持運作。它是在等待特定事件發生（例如有人透過網路連線進來）。

3. Daemon 的主要功能

Daemon 的功能非常廣泛，基本上支撐了 Ubuntu 系統的所有自動化任務：

A. 提供網絡服務

這是最常見的用途。當你架設伺服器時，Daemon 會持續監聽特定的「端口」（Port）。
例如：apache2 或 nginx Daemon 會守著 80/443 端口，當有人輸入你的網址時，它立刻響應並傳送網頁資料。

B. 管理系統硬件與資源

確保硬件與軟體之間的溝通順暢。
例如：cupsd 負責排隊管理列印任務；udevd 負責偵測你什麼時候插上了 USB。

C. 執行定時任務

有些 Daemon 專門負責「看時間」。
例如：crond。如果你設定了每天凌晨 3 點備份資料，crond 就會在那一秒準時喚醒備份程式。

D. 日誌記錄與監控

記錄系統發生了什麼事，方便管理員除錯。
例如：rsyslogd 會收集各個軟體的錯誤訊息並存成 log 檔案。

4. Ubuntu 的 systemd 架構

Ubuntu 自 15.04 版本後，全面採用 systemd 作為預設的初始化系統與服務管理器。它是所有 Daemon 的「總司令」。

systemd 層級結構圖 (Ubuntu)

PID 1 的領袖地位

當你按下 Ubuntu 的電源鍵，核心加載後第一個跑起來的就是 systemd，它的進程編號永遠是 1。它是所有其他進程的父進程。

5. 動手實作與故障修復

建立 Python 網頁守護進程

1 撰寫 Python 網頁程式

建立檔案：sudo nano /usr/local/bin/myserver.py

import http.server
import socketserver
from datetime import datetime

# 定義伺服器監聽的埠口 (Port)
PORT = 8080

class MyHandler(http.server.SimpleHTTPRequestHandler):
    """
    自定義的請求處理類別，繼承自 SimpleHTTPRequestHandler。
    """

    def do_GET(self):
        """
        當瀏覽器發送 GET 請求時自動觸發的 Function。
        這是處理網頁顯示邏輯的核心。
        """
        # 1. 向客戶端發送 HTTP 狀態碼 200，代表連線成功 (OK)
        self.send_response(200)

        # 2. 設定回應標頭 (Header)，告知瀏覽器內容是 HTML 格式且使用 UTF-8 編碼
        self.send_header("Content-type", "text/html; charset=utf-8")

        # 3. 結束標頭部分，準備開始傳送網頁本體內容
        self.end_headers()

        # 4. 取得當前的系統時間，並轉換成指定的字串格式
        now = datetime.now().strftime("%Y-%m-%d %H:%M:%S")

        # 5. 組合要顯示在網頁上的 HTML 字串內容
        content = f"<h1>Ubuntu Daemon 測試網頁</h1><p>現在時間：{now}</p><p>這是一個由 systemd 管理的 Python 守護進程。</p>"

        # 6. 將字串編碼成 bytes 並寫入傳輸管道 (wfile)，回傳給瀏覽器
        self.wfile.write(content.encode("utf-8"))

# 啟動 TCP 伺服器，監聽指定連接埠並持續運行
with socketserver.TCPServer(("", PORT), MyHandler) as httpd:
    """
    啟動伺服器並進入無限循環。
    serve_forever() 是作為 Daemon 的關鍵，它讓程式持續運行，不會執行完就結束。
    """
    httpd.serve_forever()
2 編寫 Service 檔案

建立設定檔：sudo nano /etc/systemd/system/py-web.service

[Unit]
# 服務的簡短描述，會顯示在 systemctl status 的輸出的 Description 欄位
Description=Python 簡易網頁守護進程

# 啟動順序定義：確保在「網路功能」準備就緒後，才啟動這個網頁服務
After=network.target

[Service]
# 定義啟動類型：simple 代表 ExecStart 指令執行後，即視為服務已啟動
Type=simple

# 核心啟動指令：指定使用 python3 的絕對路徑來執行你的程式碼路徑
# 注意：在服務定義檔中，所有路徑都必須使用「絕對路徑」
ExecStart=/usr/bin/python3 /usr/local/bin/myserver.py

# 自動復原機制：如果程式因為 Bug 崩潰或被意外殺掉，systemd 會自動重啟它
Restart=always

[Install]
# 定義服務要掛載在哪個「目標」下
# multi-user.target 代表「多人多工模式」，通常就是指系統開機到登入畫面後的正常運作狀態
WantedBy=multi-user.target
3 設定時區
sudo timedatectl set-timezone Asia/Taipei
4 啟動服務
sudo systemctl daemon-reload
sudo systemctl start py-web
sudo systemctl enable py-web
5 查看「所有已載入」的服務
systemctl list-units --type=service

http://[你的 IP 位址]:8080

如果 Daemon 掛掉會發生什麼事？

等級 影響範圍 掛掉的後果
輕微 單一應用功能失效 無法列印文件，但電腦其他功能正常。
嚴重 服務中斷 無法透過 SSH 連線到這台伺服器。
災難 系統不穩定或停擺 系統可能直接卡死，因為 systemd 是所有進程的「父親」。

如何徹底刪除服務 (Uninstall Service)

第一步：停止並禁用服務
sudo systemctl stop py-web
sudo systemctl disable py-web
第二步：刪除服務定義檔
sudo rm /etc/systemd/system/py-web.service
第三步：重載系統管理員 (最重要！)
sudo systemctl daemon-reload
第四步：清理殘留 (選用)
sudo systemctl reset-failed

如何修復掛掉的 Daemon？

第一步：確認狀態 (Status)

sudo systemctl status py-web

第二步：重設失敗狀態 (Reset Failed)

先清除該服務之前殘留的「失敗」標記，這能確保重啟時狀態是乾淨的。

sudo systemctl reset-failed

第三步：重啟服務 (Restart)

sudo systemctl restart py-web

第四步：檢查錯誤日誌 (Journal)

sudo journalctl -u py-web -xe

進階：讓系統自動幫你修復

[Service]
Restart=always
RestartSec=5s

6. Daemon 的製造原理：先「細胞分裂」再「靈魂奪舍」

在 Linux 的世界裡，除了第一個進程（PID 1 的 systemd）是 Kernel 伸手拉起來的，其他的進程都必須透過「父傳子」的方式產生。

第一步：fork (影分身 / 細胞分裂)

  • 1. 發起請求：父進程 (systemd) 向 Kernel 發起系統呼叫。
  • 2. 核心運作：Kernel 複製一份跟父進程一模一樣的副本，包括資源與記憶體空間。
  • 結果：出現一個擁有全新 PID (如 4567) 的子進程副本。

第二步：exec (奪舍 / 轉生)

  • 3. 環境整備：子進程處理 chdir (切換目錄)、setuid (權限降級) 等設定。
  • 4. 核心運作：執行 execve()，清空記憶體，載入新程式。
  • 結果：一個嶄新的 Daemon 靈魂在原有的 PID 容器中降生。

7. Alpine Linux 的 Daemon

在理解了 Ubuntu 的 Daemon 管理與製造原理後，我們來看看專為極致精簡設計的 Alpine Linux。

❓ 為什麼架構圖裡沒有 systemd？

Alpine Linux 不使用 systemd。它選擇使用傳統且輕量的 OpenRC 作為管理器，以維持 5MB 的極小體積。
